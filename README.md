# Mini Library Management System

## Project title
Mini Library Management System

## Short description
This project is a console-based Library Management System written in C++. It supports managing books, registering users, logging in as members or librarians, issuing and returning books, tracking loans, and calculating overdue fines. The project demonstrates core object-oriented programming concepts including encapsulation, inheritance, abstraction, polymorphism, operator overloading, templates, and exception handling.

## How to compile and run
Compile from the project folder with:

```bash
g++ -std=c++17 -Wall -Wextra -pedantic MiniLibrary.cpp Book.cpp BookCatalog.cpp Fine.cpp LibrarySystem.cpp Loan.cpp ProjectExceptions.cpp TransactionManager.cpp -o mini_library
```

Run with:

```bash
./mini_library
```

On Windows (MinGW), run with:

```bash
mini_library.exe
```

## Sample input files
This project uses plain text files for persistence.

Required files:
- `books.txt`
- `loans.txt`
- `users.txt`

Place all three files in the same directory as the executable before running the program.

### File conventions
- `books.txt` stores serialized `Book` records in this format:
  ```
  BookID|Title|Author|AvailabilityFlag
  ```
  Example:
  ```
  B001|Clean Code|Robert C. Martin|1
  ```

- `loans.txt` stores serialized `Loan` records in this format:
  ```
  BookID|MemberUsername|IssueDate|DueDate|ReturnDate
  ```
  If a book has not yet been returned, `ReturnDate` is stored as `-`.
  Example:
  ```
  B001|alice|2026-03-20|2026-04-03|-
  ```

- `users.txt` stores serialized `UserAccount` records.
  Member format:
  ```
  MEMBER|username|passwordHash|Member|name|age|memberId
  ```
  Librarian format:
  ```
  LIBRARIAN|username|passwordHash|Librarian|name|age|staffId
  ```

If the files are missing or cannot be opened, the program handles that through file-related exceptions and continues with warnings where possible.

## Relationships

### Ownership / usage relationships
- `BookCatalog` owns and manages a `Repository<Book>`.
- `TransactionManager` owns and manages a `Repository<Loan>`.
- `Librarian` uses `Repository<Book*>` to track managed books.
- `Librarian` uses `Repository<Fine*>` to track collected fines.
- `MiniLibrary.cpp` creates and coordinates `BookCatalog`, `TransactionManager`, and `Repository<UserAccount*>`.
- `TransactionManager` uses `Book` and `Member` objects to issue and return books.
- `Loan` stores references logically by identifiers (`bookId`, `memberUsername`) instead of direct object ownership.

### Inheritance relationships
- `Fine` inherits from `Payable`.
- `Member` and `Librarian` inherit from both `Person` and `UserAccount`.

### Why inheritance is used
- `Payable` provides a common payment-related interface.
- `Person` stores shared identity information for users.
- `UserAccount` stores shared authentication behavior.
- `Member` and `Librarian` specialize behavior such as login messages, serialization, and role-based actions.

## How polymorphism is applied
Polymorphism is used in several ways:

1. **Runtime polymorphism through abstract base classes**
   - `Payable*` can refer to a `Fine` object.
   - `UserAccount*` can refer to either a `Member` or a `Librarian`.
   - `Person*` can refer to derived classes and call overridden `displayInfo()`.

2. **Method overriding**
   - `Fine` overrides `processPayment()` and `getPaymentDetails()` from `Payable`.
   - `Member` and `Librarian` override `login()`, `logout()`, `displayInfo()`, and `serialize()`.

3. **Template polymorphism**
   - `Repository<T>` works with multiple object types such as `Book`, `Loan`, `UserAccount*`, `Book*`, and `Fine*`.

## Exception Handling
The project expects and handles these exceptions:

### 1. `FileAccessException`
Thrown when a data file cannot be opened for reading or writing.

**Handled in:**
- startup loading logic in `main()`
- shutdown saving logic in `main()`
- member registration save logic

### 2. `UsernameTakenException`
Thrown during registration if the requested username already exists.

**Handled in:**
- `registerNewMember()`

### 3. `LoginFailedException`
Thrown when a login attempt uses invalid credentials.

**Handled in:**
- login section of `main()`

### 4. `DuplicateLoanException`
Thrown when a member tries to borrow the same book again while that loan is still active.

**Handled in:**
- member borrow flow after search
- direct borrow-by-ID flow

### General handling approach
- Errors that should stop only one action are caught locally and shown to the user.
- Errors that affect file loading/saving are caught in `main()` and shown as warnings.
- The program attempts to continue running whenever recovery is possible.

## Known issues
- `Librarian` stores raw pointers to managed books and collected fines, so lifetime management depends on the rest of the program remaining consistent.
- The project uses simple text serialization, so malformed input lines may still cause incorrect deserialization behavior.
- Password hashing uses a simple custom hash and is not suitable for real-world security.
- Dates are stored as strings and handled with basic time functions, which is acceptable for a small course project but not ideal for production software.
- `Repository<T>` does not automatically delete owned pointer elements, so memory management must be handled carefully by the program.
- Input validation for user-entered ages and some menu inputs is basic.
